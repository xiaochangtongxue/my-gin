## 二、子任务详细方案

### 5.1 Swagger 文档（swaggo/swag）

**目标**：集成 Swagger，自动生成 API 文档，支持在线调试。

#### 技术选型
- **swaggo/swag** - Swagger 文档生成工具
- **swaggo/gin-swagger** - Gin 中间件
- **swaggo/files** - 静态文件托管

#### 实现方案

**5.1.1 安装依赖**
```bash
go install github.com/swaggo/swag/cmd/swag@latest
go get -u github.com/swaggo/gin-swagger
go get -u github.com/swaggo/files
```

**5.1.2 项目结构**
```
my-gin/
├── docs/                     # Swagger 文档生成目录
│   ├── docs.go              # 自动生成
│   ├── swagger.json         # 自动生成
│   └── swagger.yaml         # 自动生成
├── cmd/server/
│   └── main.go              # 添加 Swagger 总体注解
├── internal/handler/         # Handler 添加接口注解
└── pkg/response/
    └── response.go          # 定义通用响应结构用于 Swagger
```

**5.1.3 main.go 添加总体注解**
```go
// @title           My-Gin API
// @version         1.0
// @description     生产级 Gin 框架脚手架 API
// @termsOfService  http://swagger.io/terms/

// @contact.name   API Support
// @contact.url    http://www.example.com/support
// @contact.email  support@example.com

// @license.name  Apache 2.0
// @license.url   http://www.apache.org/licenses/LICENSE-2.0.html

// @host      localhost:8080
// @BasePath  /api/v1

// @securityDefinitions.apikey BearerAuth
// @in header
// @name Authorization
// @description Type "Bearer" followed by a space and JWT token.
```

**5.1.4 Handler 注解规范**
```go
// @Summary      简短描述
// @Description  详细描述
// @Tags         分组标签
// @Accept       json
// @Produce      json
// @Param        id   path      int     true  "ID"
// @Param        request body req.CreateUserReq true "请求体"
// @Success      200  {object}  response.Response{data=resp.UserResp}
// @Failure      400  {object}  response.Response
// @Failure      401  {object}  response.Response
// @Router       /users/{id} [get]
// @Security     BearerAuth
```

**5.1.5 通用响应结构定义**
需要在 `pkg/response/` 中定义 Swagger 使用的响应模型：
```go
// Response 统一响应结构（Swagger 注解用）
// @Description 统一 API 响应格式
type Response struct {
    Code      int         `json:"code" example:"0"`
    Message   string      `json:"message" example:"success"`
    Data      interface{} `json:"data"`
    Timestamp int64       `json:"timestamp" example:"1700000000"`
    RequestID string      `json:"request_id" example:"abc123"`
}
```

**5.1.6 路由注册**
```go
import swaggerFiles "github.com/swaggo/files"
ginSwagger "github.com/swaggo/gin-swagger"

// 在路由中注册
engine.GET("/swagger/*any", ginSwagger.WrapHandler(swaggerFiles.Handler))
```

**5.1.7 生成文档命令**
```bash
# 在项目根目录执行
swag init -g cmd/server/main.go -o docs
```

### 5.2 健康检查增强

**目标**：提供完整的健康检查接口，检查数据库、Redis 等依赖状态。

#### 当前状态
- 已有基础 `/health` 接口，仅返回固定 OK 状态

#### 实现方案

**5.2.1 新建健康检查 Handler**
```
internal/handler/health.go
```

**5.2.2 健康检查响应结构**
```go
type HealthResponse struct {
    Status    string            `json:"status"`     // healthy, degraded, unhealthy
    Timestamp string            `json:"timestamp"`
    Version   string            `json:"version"`
    Checks    map[string]Check  `json:"checks"`
}

type Check struct {
    Status string `json:"status"`  // up, down, degraded
    Error  string `json:"error,omitempty"`
    Latency string `json:"latency"`  // 响应时间，如 "5ms"
}
```

**5.2.3 检查项**
| 检查项 | 说明 | 超时时间 |
|--------|------|----------|
| database | MySQL 连接状态 | 2s |
| redis | Redis 连接状态 | 1s |
| server | 服务器状态 | - |

**5.2.4 路由**
- `GET /health` - 简化版健康检查（仅返回 status）
- `GET /health/ready` - 就绪检查（完整检查项）
- `GET /health/live` - 存活检查（仅检查关键依赖）

### 5.3 Prometheus Metrics

**目标**：集成 Prometheus 指标收集，暴露 `/metrics` 端点。

#### 技术选型
- **prometheus/client_golang** - Prometheus 官方 Go 客户端

#### 实现方案

**5.3.1 新建 Metrics 包**
```
pkg/metrics/
├── metrics.go       # 指标定义和收集
└── middleware.go    # HTTP 请求指标收集中间件
```

**5.3.2 指标类型**

| 指标名称 | 类型 | 说明 | 标签 |
|----------|------|------|------|
| `http_requests_total` | Counter | HTTP 请求总数 | method, path, status |
| `http_request_duration_ms` | Histogram | 请求耗时分布 | method, path |
| `http_requests_in_flight` | Gauge | 当前处理中的请求数 | - |
| `db_connections_active` | Gauge | 数据库活跃连接数 | - |
| `db_connections_idle` | Gauge | 数据库空闲连接数 | - |
| `redis_commands_total` | Counter | Redis 命令总数 | command, status |
| `redis_request_duration_ms` | Histogram | Redis 请求耗时 | command |

**5.3.3 中间件集成**
```go
// 在 HTTP 请求处理中自动收集指标
func MetricsMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()

        // 处理请求
        c.Next()

        // 记录指标
        duration := time.Since(start).Milliseconds()
        httpRequestsTotal.WithLabelValues(
            c.Request.Method,
            c.FullPath(),
            strconv.Itoa(c.Writer.Status()),
        ).Inc()

        httpRequestDuration.Observe(float64(duration))
    }
}
```

**5.3.4 路由注册**
```go
import "github.com/prometheus/client_golang/prometheus/promhttp"

// 注册 /metrics 端点（跳过认证和日志记录）
engine.GET("/metrics", gin.WrapH(promhttp.Handler()))
```

**5.3.5 配置项**
在 `config.yaml` 中添加：
```yaml
metrics:
  enable: true              # 是否启用 metrics
  path: "/metrics"          # metrics 端点路径
  skip_paths:               # 跳过统计的路径
    - "/health"
    - "/metrics"
```

## 四、开发顺序建议

1. **Swagger 文档**（优先级高）
   - 安装 swag 工具
   - 添加 main.go 总体注解
   - 完善 Handler 接口注解
   - 集成 gin-swagger 路由
   - 生成文档并测试

2. **健康检查增强**（优先级高）
   - 创建 Health Handler
   - 实现数据库/Redis 健康检查
   - 创建 Health Router
   - 更新 main.go 路由注册

3. **Prometheus Metrics**（优先级中）
   - 创建 pkg/metrics 包
   - 定义指标和中间件
   - 集成到 Gin 中间件链
   - 暴露 /metrics 端点
