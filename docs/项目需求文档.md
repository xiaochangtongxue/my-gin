## 二、模块详细设计

### 模块 1: 配置管理（pkg/config）

**功能描述**:
提供统一的配置管理方案，支持多环境配置、环境变量覆盖、配置热加载。

#### 子功能 1.1: 配置文件管理
- **功能描述**: 支持多环境配置文件加载（dev/test/prod），支持 YAML/JSON/TOML 格式
- **所需技术**: Viper
- **实现方案**:
  - 定义配置结构体（Server、Database、Redis、Logger、JWT 等）
  - 根据环境变量或运行模式加载对应配置文件
  - 配置文件路径支持命令行参数覆盖
  - 配置变更时触发回调函数

#### 子功能 1.2: 环境变量支持
- **功能描述**: 支持通过环境变量覆盖配置项，符合 12-Factor App 规范
- **所需技术**: Viper AutomaticEnv
- **实现方案**:
  - 自动绑定环境变量到配置结构
  - 支持环境变量前缀（如 APP_）
  - 支持嵌套配置的环境变量映射

#### 子功能 1.3: 配置校验
- **功能描述**: 启动时校验配置合法性，防止错误配置导致运行时异常
- **所需技术**: Go Validator
- **实现方案**:
  - 在配置结构体上添加校验标签
  - 启动时进行校验，失败则 panic
  - 提供友好的错误提示

---

### 模块 2: 日志系统（pkg/logger）

**功能描述**:
提供高性能的结构化日志系统，支持日志轮转、分级日志、链路追踪。

#### 子功能 2.1: 结构化日志
- **功能描述**: 使用 Zap 提供高性能结构化日志，支持 JSON 和 Console 格式
- **所需技术**: Uber Zap
- **实现方案**:
  - 封装 Zap 日志库，提供 Info/Error/Debug/Fatal 等方法
  - 支持日志级别动态配置
  - 支持字段化日志，便于查询和分析
  - 同时输出到文件和控制台

#### 子功能 2.2: 日志轮转
- **功能描述**: 支持日志文件按大小、时间自动轮转和压缩归档
- **所需技术**: Lumberjack
- **实现方案**:
  - 配置最大文件大小（如 500MB）
  - 配置保留天数（如 90 天）
  - 配备保留文件最大数量
  - 自动压缩旧日志文件

#### 子功能 2.3: 请求日志追踪
- **功能描述**: 为每个请求生成唯一 RequestId，贯穿整个请求生命周期，便于问题追踪
- **所需技术**: Gin 中间件、上下文传递
- **实现方案**:
  - 请求入口生成 RequestId（UUID 或 Snowflake）
  - 通过 Gin Context 传递 RequestId
  - 日志中自动记录 RequestId
  - 响应 Header 中返回 RequestId

#### 子功能 2.4: 链路追踪集成
- **功能描述**: 集成 OpenTelemetry，支持分布式链路追踪
- **所需技术**: OpenTelemetry Go
- **实现方案**:
  - 集成 OTel SDK
  - 自动记录 HTTP 请求 Span
  - 支持导出到 Jaeger/Zipkin
  - 记录数据库查询 Span

### 模块 3: 数据库层（pkg/database + internal/repository）

**功能描述**:
提供数据库连接池管理、ORM 封装、事务管理、读写分离。

#### 子功能 3.1: 数据库连接管理
- **功能描述**: 管理 MySQL 连接池，支持连接复用和健康检查
- **所需技术**: GORM
- **实现方案**:
  - 初始化 GORM DB 实例
  - 配置连接池参数（最大连接数、空闲连接数、连接最大生命周期）
  - 实现 Ping 方法检查连接健康状态
  - 支持慢查询日志记录

#### 子功能 3.2: 事务管理
- **功能描述**: 提供事务管理封装，支持嵌套事务和事务回滚
- **所需技术**: GORM Transaction
- **实现方案**:
  - 提供 Transaction 方法封装
  - 支持 context.Context 传递事务
  - 自动处理事务提交和回滚
  - 支持嵌套事务（SavePoint）

#### 子功能 3.3: 数据库迁移
- **功能描述**: 提供数据库版本管理和迁移能力
- **所需技术**: golang-migrate
- **实现方案**:
  - 创建迁移文件目录
  - 提供 Up/Down 迁移脚本
  - 支持命令行执行迁移
  - 记录迁移历史

#### 子功能 3.4: 读写分离
- **功能描述**: 支持主从读写分离，提升数据库并发能力
- **所需技术**: GORM Plugin
- **实现方案**:
  - 配置主库和从库连接
  - 写操作自动路由到主库
  - 读操作路由到从库（负载均衡）
  - 主库宕机自动降级

### 模块 4: 缓存层（pkg/cache）

**功能描述**:
提供统一的缓存抽象层，支持 Redis 和内存缓存。

#### 子功能 4.1: Redis 缓存
- **功能描述**: 封装 Redis 客户端，提供常用的缓存操作方法
- **所需技术**: go-redis
- **实现方案**:
  - 初始化 Redis 客户端
  - 提供 Get/Set/Del 等基础方法
  - 支持过期时间设置
  - 支持序列化和反序列化
  - 连接池管理

#### 子功能 4.2: 本地缓存
- **功能描述**: 提供进程内内存缓存，适用于热点数据
- **所需技术**: go-cache 或 BigCache
- **实现方案**:
  - 封装本地缓存库
  - 支持 LRU 淘汰策略
  - 支持过期时间
  - 提供 Get/Set/Del 方法

#### 子功能 4.3: 缓存抽象接口
- **功能描述**: 定义缓存接口，支持多种缓存实现切换
- **所需技术**: Go Interface
- **实现方案**:
  - 定义 Cache 接口（Get/Set/Del/Exists）
  - RedisCache 和 MemoryCache 实现接口
  - 支持通过配置选择缓存实现
  - 支持缓存穿透、击穿、雪崩防护

### 模块 5: 统一响应处理（pkg/response）

**功能描述**:
提供统一的 API 响应格式，标准化成功和失败响应。

#### 子功能 5.1: 标准响应格式
- **功能描述**: 定义统一的 JSON 响应格式
- **所需技术**: Gin Context
- **实现方案**:
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {},
    "timestamp": 1700000000,
    "request_id": "xxx"
  }
  ```
  - 提供 Success、Error 方法
  - 自动添加时间戳和 RequestId
  - 支持泛型

#### 子功能 5.2: 错误码定义
- **功能描述**: 定义全局错误码常量和错误消息
- **所需技术**: Go Const
- **实现方案**:
  - 定义标准 HTTP 错误码（400/401/403/404/500）
  - 定义业务错误码（10000-19999）
  - 支持国际化错误消息
  - 提供错误码文档

#### 子功能 5.3: 分页响应
- **功能描述**: 封装分页数据响应格式
- **所需技术**: Go Struct
- **实现方案**:
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {
      "items": [],
      "total": 100,
      "page": 1,
      "page_size": 10
    }
  }
  ```
  - 提供 PageResponse 方法
  - 自动计算总页数
  - 支持游标分页（可选）

### 模块 6: 错误处理（pkg/errors）

**功能描述**:
提供全局错误处理机制，区分业务错误和系统错误。

#### 子功能 6.1: 业务错误定义
- **功能描述**: 定义业务错误类型，支持错误码和错误消息
- **所需技术**: Go Error Interface
- **实现方案**:
  - 定义 BusinessError 结构体
  - 实现 Error() 方法
  - 支持错误码和错误消息
  - 支持错误堆栈

#### 子功能 6.2: 全局错误恢复
- **功能描述**: 捕获所有 panic 和错误，防止程序崩溃
- **所需技术**: Gin Recovery Middleware
- **实现方案**:
  - 实现 Recovery 中间件
  - 捕获 panic 并记录错误日志
  - 返回友好的错误响应
  - 包含堆栈信息（仅开发环境）

#### 子功能 6.3: 错误日志记录
- **功能描述**: 自动记录错误日志，包含完整的上下文信息
- **所需技术**: Zap
- **实现方案**:
  - 在错误处理中间件中记录
  - 记录 RequestId、错误堆栈、请求参数
  - 区分错误级别（Warn/Error/Fatal）
  - 支持错误告警（接入钉钉/企业微信）

### 模块 7: 参数验证（pkg/validator）

**功能描述**:
提供请求参数验证，支持自定义验证规则和国际化错误消息。

#### 子功能 7.1: 参数绑定和验证
- **功能描述**: 自动绑定请求参数到结构体并进行验证
- **所需技术**: Gin Binding、Go Validator
- **实现方案**:
  - 使用 ShouldBindJSON/Query 等方法
  - 在 DTO 结构体上添加 validate 标签
  - 验证失败返回详细的错误信息
  - 支持自定义验证消息

#### 子功能 7.2: 自定义验证规则
- **功能描述**: 支持自定义验证函数
- **所需技术**: Go Validator Custom Functions
- **实现方案**:
  - 注册自定义验证器
  - 示例：手机号验证、身份证验证、密码强度验证
  - 支持正则表达式验证
  - 验证器可复用

#### 子功能 7.3: 国际化错误消息
- **功能描述**: 支持多语言验证错误消息
- **所需技术**: Go Validator Translations
- **实现方案**:
  - 注册多语言翻译器
  - 根据请求头 Accept-Language 选择语言
  - 支持中文、英文等
  - 错误消息可配置

### 模块 8: 认证授权（pkg/jwt + internal/middleware）

**功能描述**:
提供 JWT 认证和 RBAC 权限控制。

#### 子功能 8.1: JWT 认证
- **功能描述**: 基于 JWT 的用户认证机制

### 模块 9: 限流与保护（pkg/limiter + internal/middleware）

**功能描述**:
提供限流、熔断等保护机制，防止服务过载。

#### 子功能 9.1: 限流器
- **功能描述**: 基于 IP 或用户的请求限流
- **所需技术**: Redis + Lua、Golang Rate
- **实现方案**:
  - 滑动窗口算法
  - 令牌桶算法
  - 支持 IP 限流（防 DDoS）
  - 支持用户限流（防刷接口）
  - 支持 API 级别限流
  - 超出限制返回 429 状态码

#### 子功能 9.2: 熔断器
- **功能描述**: 当下游服务不可用时自动熔断，防止雪崩
- **所需技术**: Hystrix-Go 或 Sentinel-Golang
- **实现方案**:
  - 配置熔断阈值（错误率、响应时间）
  - 达到阈值自动触发熔断
  - 熔断后自动降级处理
  - 支持半开启状态探测恢复

### 模块 10: 安全防护（internal/middleware）

#### 子功能 10.3: CSRF 防护
- **功能描述**: 防止跨站请求伪造攻击
- **所需技术**: CSRF Token

  - 生成 CSRF Token

  - 请求时验证 Token
  - 支持双重 Cookie 验证

#### 子功能 10.4: SQL 注入防护
- **功能描述**: 防止 SQL 注入攻击
- **所需技术**: GORM Parameterized Query
- **实现方案**:
  - 使用参数化查询
  - 禁止拼接 SQL
  - 使用 GORM ORM 框架
  - 输入参数过滤

#### 子功能 10.5: 安全响应头
- **功能描述**: 添加安全相关的 HTTP 响应头
- **所需技术**: Gin Middleware
- **实现方案**:
  - X-Frame-Options: DENY
  - X-Content-Type-Options: nosniff
  - X-XSS-Protection: 1; mode=block
  - Strict-Transport-Security
  - Content-Security-Policy

## 六、开发规范

### 代码规范
- 遵循 `gofmt` 格式化
- 遵循 `Effective Go` 最佳实践
- 使用 `golangci-lint` 进行代码检查
- 函数注释使用 Godoc 格式
- 包名使用小写单词
- 接口名以 -er 结尾
- 避免代码中存在魔法值

### Git 规范
- 分支命名: `feature/`、`bugfix/`、`hotfix/`
- Commit Message: 遵循 Conventional Commits 规范
- Code Review: 必须 Code Review 后才能合并
- 版本标签: 使用语义化版本（Semantic Versioning）

### 测试规范
- 单元测试覆盖率 > 80%
- 关键业务逻辑必须有测试
- 使用 Mock 隔离依赖
- 表格驱动测试
